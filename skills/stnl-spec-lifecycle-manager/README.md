# stnl-spec-lifecycle-manager

This package maintains independent feature SPECs. It covers documentary requirements and their lifecycle, not implementation planning or delivery operations.

The four modes are `INIT`, `RESUME`, `READINESS`, and `CLOSE`. `READINESS` never mutates the SPEC workspace and supports `LOCAL` or `GLOBAL`; a `GLOBAL/READY` verdict is bound to its lifecycle-authority snapshot by an ephemeral external readiness attestation required by `CLOSE`. Mutable modes validate an isolated candidate before publication. Canonical records are directly rendered Markdown whose heading is the sole ID authority; records are retired in place rather than physically removed. YAML is limited to file-purpose and compact feature-level discovery/state blocks.

Supporting resources are progressively loaded from `references/`, `templates/`, `examples/`, and `evals/`. Validate changes with `python3 scripts/test-spec-lifecycle.py` from the repository root.

The publisher keeps `.<workspace-name>.lifecycle.lock` beside each workspace as persistent runtime metadata. It coordinates cooperating publishers and is deliberately not unlinked between operations. The lock is outside the workspace, is not SPEC authority, and is excluded from readiness and canonical snapshots, candidate copying, rendering, and closed output. Repositories embedding workspaces should ignore it with `.*.lifecycle.lock`; this repository provides that rule in `.gitignore`. Transaction journals, stages, backups, manifests, and readiness attestations are not locks and must be cleaned rather than ignored.

Publisher target, candidate, manifest, and attestation paths reject traversal and every non-system symlink component; existing components are canonicalized to their physical directory-entry spelling, so case and NFC/NFD aliases cannot bypass disjointness, containment, or target-owned runtime-namespace checks. Input namespace checks complete before lock creation. Source and stage digests are captured before validation and rechecked afterward; a CLOSE stage also rejects OS metadata absent from deterministic rendering. After `target` is renamed to its transaction backup, the publisher hashes that backup before candidate promotion. The canonical snapshot includes bytes, entry types, link counts, and relative hardlink peers. A mismatch restores the exact captured state to the official path, validates there, aborts, and cleans safe transaction residue. This covers changes visible to those digest checks; it does not claim filesystem-wide atomicity or detection of writes that occur after verification.

A readiness attestation is strict JSON with exactly `version`, `mode`, `scope`, `verdict`, `workspace_identity`, and `workspace_snapshot_sha256`; identity contains only the canonical H1 and a SHA-256 of the normalized workspace path. Its snapshot domain contains the raw path and bytes of `feature_spec.md` plus every indexed, materialized `shared/` authority file. It excludes execution, code, OS packaging metadata, publisher runtime metadata, manifests, and attestations. The deterministic creator accepts only `GLOBAL/READY`, writes outside the workspace, and rejects non-system symlink components and linked authority files. Renderer and publisher both reject wrong identity or stale authority; the publisher additionally requires the candidate to equal the deterministic rendering of that live attested source. The attestation is deleted after terminal CLOSE success and never becomes durable authority.
